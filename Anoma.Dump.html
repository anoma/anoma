        Anoma.Dump (Anoma v0.17.0)

        I provide an interface to dump current state and load appropriate external files to launch them as Anoma nodes.

        You can also use me to dump info such as current states and tables in a readable map format as well as get info stored in external files in binary format.

        Dumping API

        I give access to following public dumping functionality:

        • dump/2
        • get_all/1
        • get_state/1
        • get_tables/1

        Loading API

        I give access to following public loading functionality

        • launch/2
        • launch/3
        • load/1

        dump(name, node)

        @spec dump(Path.t(), atom()) :: {:ok, :ok} | {:error, any()}

        I dump the current state with storage. I accept a string as a name, so that the resulting file will be created as name.txt in the appropriate data directory. As a second argument I accept a node name whose info presented as a map I dump as a binary.

        Note that if the environment is test we do not use the XDG format for storing data and instead dump the files in the immadiate app folder.

        The map typing can be seen in get_all

        launch(file, name, options \\ [])

        @spec launch(String.t(), atom(), [launch_option()]) ::
  {:ok,
   %Anoma.Node{
     clock: term(),
     configuration: term(),
     dumper: term(),
     executor: term(),
     executor_topic: term(),
     logger: term(),
     logger_topic: term(),
     mempool: term(),
     mempool_topic: term(),
     ordering: term(),
     pinger: term(),
     router: term(),
     storage: term(),
     storage_topic: term(),
     transport: term()
   }}
  | any()

        I launch a node given a file containing a binary version of an 12-tuple with appropriate info in the following order:

        • router id
        • mempool topic id
        • executor topic id
        • dumper
        • storage
        • logger
        • clock
        • ordering
        • mempool
        • pinger
        • executor
        • storage names
        • qualified
        • order
        • block_storage

        All engines have info on their states and id's so that checkpointing the system will keep all adresses used in the previous session. Note that I ensure that the apporpriate tables are new.

        Moreover, I ensure that the mempool and block storage are in sync. In particular, I check that the order of the last block is less than that of the mempool dumped. If not, I manually remove the last block.

        Check whether your transactions have had an assigned worker. If not, relaunch them directly. If blocks were out of sync with mempool, relaunch the executions as well.

        Options

        see launch_options/0 for the full list of optional arguments

        load(name)

        @spec load(String.t()) :: any() | dump()

        I read the given file which I assume contains binary info and convert it to an Elixir term.

        As the dumped state may have extra atoms not present in the session, I currently allow for atom creation in the loaded term.
